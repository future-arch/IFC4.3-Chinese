工业基础类（IFC）是建筑资产行业数据共享的开放国际标准。该标准包括：

  1. 模式（以多种形式提供，参见范围）
  2. 文档（以HTML格式提供，使用Markdown编写）
  3. 属性和数量集定义（在模式中实现的可扩展性机制的标准化定义 - 以XML格式提供并在HTML文档中可用）
  4. 数据文件的交换或序列化机制，参见范围

IFC 最常用的交换格式是基于使用EXPRESS语言（ISO 10303-11:2004）发布的模式的Step物理文件格式（ISO 10303-21:2002）。它是构成交换的实体实例的明文编码，其中属性值以未命名值的有序序列形式提供。

约定

IFC 规范包含源自建筑和设施管理行业领域、专业和行业使用的术语、概念和数据规范项。术语和概念使用简单的英语单词，数据规范中的数据项遵循命名约定。

  • 类型、实体、规则和函数的数据项名称以 "Ifc" 前缀开头，并继续使用驼峰命名法的英语单词（无下划线，单词首字母大写）；
  • 实体内的属性名称遵循驼峰命名法，无前缀；
  • 作为本标准一部分的属性集定义以 "Pset_" 前缀开头，并继续使用驼峰命名法的英语单词；
  • 作为本标准一部分的数量集定义以 "Qto_" 前缀开头，并继续使用驼峰命名法的英语单词。

buildingSMART International 在 translations.buildingsmart.org 上发布这些术语和概念的其他人类语言翻译。

这三个 MVD 可以看作是 IFC 4.3 的三个实施级别。它们是逐步增加实施中更高级功能的级别。供应商可以针对这些 MVD 获得 IFC 导入认证。导出认证可以针对 MVD 进行，也可以针对更细粒度的功能部分进行。这些可以在 validate.buildingsmart.org 上找到。 IFC 文档存放在 standards.buildingsmart.org。

架构

IFC 的数据模式架构定义了四个概念层，每个单独的模式都恰好分配到一个概念层。下图显示了 IFC 分层架构的模式架构。

  1. 资源层 — 最底层包括所有包含资源定义的单独模式，这些定义不包括全局唯一标识符，不应独立于在更高层声明的定义使用；
  2. 核心层 — 下一层包括内核模式和核心扩展模式，包含最一般的实体定义，在核心层或以上定义的所有实体都带有全局唯一 ID 和可选的拥有者和历史信息；
  3. 互操作性层 — 下一层包括包含跨多个学科使用的通用产品、过程或资源专业化的实体定义的模式，这些定义通常用于跨领域交换和共享建筑信息；
  4. 领域层 — 最高层包括包含特定于某个学科的产品、过程或资源专业化的实体定义的模式，这些定义通常用于领域内交换和共享信息。 在此图中，"隧道"领域是透明的，因为这不是当前 IFC 4.3 的一部分，但旨在包含在未来的更新中。

兼容性

建筑资产具有很长的使用寿命。IFC 版本之间的兼容性是标准开发中的一个关键关注点。

向后兼容性是指针对标准先前版本编写的数据文件能够被支持后续版本的应用程序读取的能力。

本国际标准的 ISO 10303-11 EXPRESS 模式分发中的类通常反映在程序代码中，并直接影响使用 ISO 10303-21 交换格式的交换的结构（属性计数、属性类型）。

相反，属性和数量集通常是更补充性的数据，不会在相同程度上影响软件接口的功能。属性和数量集也明确提供，以快速解决特定用例和不断发展的软件能力。属性和数量集的更改不会影响 ISO 10303-21 交换格式的结构（属性计数、属性类型）。

因此，属性和数量集的兼容性标准与 EXPRESS 模式的兼容性标准不同。

除了极少数情况外，向后兼容性是 IFC 标准演进中的一个硬性要求。然而，可能发生的更改类型在严重程度上有所不同。例如，重命名实体名称或在属性序列的开头插入属性会导致解析针对标准先前版本编写的文件时出现几乎不可恢复的错误。而向现有枚举添加枚举项被认为是完全向后兼容的。

注释 本版本的开发未明确考虑向前兼容性，但在可能的情况下被视为可取的功能。

弃用

随着 IFC 标准的发展，某些构造变得不受欢迎，因为它们已被其他构造取代，因为软件实现被证明不足，或者因为在定义中发现了无法以向后兼容的方式纠正的缺陷。

在这种情况下，通常选择弃用该构造，给实现者充足的时间来调整他们的软件。

注释 1 符合要求的解释器仍应能够导入已弃用的定义。
注释 2 符合要求的解释器应考虑修改导出，使用建议的替代定义而不是已弃用的定义。
